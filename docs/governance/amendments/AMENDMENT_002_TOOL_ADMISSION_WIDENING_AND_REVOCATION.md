# Amendment 002: Tool Admission, Widening, and Revocation

## Purpose

This amendment defines the minimum governance path for admitting, widening, reviewing, and revoking tools that materially change Founderos capability.

## Admission rule

A tool is not part of Founderos merely because it can be installed or invoked.

A tool becomes constitutionally admitted only when:
1. its purpose is documented,
2. its power class is documented,
3. its reachable domain is documented,
4. its credential placement is documented,
5. its review expectations are documented,
6. its logging expectations are documented,
7. and its admission is reflected in governance-bearing docs or decisions.

## Widening rule

A tool that already exists may still require new governance when its practical power expands.

Widening should be presumed governance-bearing when it changes one or more of the following:
- reachable external domains,
- write authority,
- workspace breadth,
- autonomy of invocation,
- durability of effects,
- operational routing authority,
- or consequences of failure.

## Review rule

Tools that affect external systems, durable records, workspace state, identity, money, publishing, deployments, or execution authority should be treated as review-bearing and normally amendment-bearing.

## Revocation rule

Every admitted tool should have a practical revocation path, such as:
- token revocation,
- plugin disablement,
- config removal,
- scope restriction,
- workflow suspension,
- or amendment-level withdrawal or replacement.

## Failure rule

If a tool behaves outside scope, loses legibility, creates unreviewable action, or erodes authority separation, the safer response is suspension first and redesign second.

## Interpretation rule

Convenience is not sufficient reason to widen a tool’s scope.

A wider power should require new justification, not merely successful prior use.
