# Amendment 004: Project Scope and Context Signaling

## Purpose

This amendment formalizes a simple two-level work model for Founderos:

- repo scope as the default work context
- project scope as the only narrower work context beneath repo scope

It also formalizes a stable signaling rule for project-scoped communication.

## Principle

Scope should reduce ambiguity, not multiply it.

Founderos should prefer a small number of explicit work scopes over a large hierarchy of nested contexts.

## Rule

Founderos recognizes only two stable work scopes unless a later amendment states otherwise:

1. **Repo scope**
2. **Project scope**

Repo scope is the default starting scope for new threads and ordinary discussion.

Project scope is entered only when a bounded project is explicitly active.

## Context signaling rule

When operating inside a project, project-related messages should begin by clearly signaling the active project name in bold.

This signal exists to preserve project awareness for both founder and system.

Outside project scope, messages should remain normal.

## Intent

This amendment exists to keep Founderos simple while making project work safer and more legible.

It is not intended to create a large project-management hierarchy inside the repo.
It is intended to create one clear narrower context beneath repo work and no more.

## Relationship to protected instruction surfaces

Protected GPT instruction files may later be updated to reflect this rule directly.
Until then, the rule remains part of the public governance and principles corpus and should be treated as the intended operating model.
