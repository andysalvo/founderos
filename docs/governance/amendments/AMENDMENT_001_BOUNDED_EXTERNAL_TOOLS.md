# Amendment 001: Bounded External Tools

## Purpose

This amendment creates the first constitutional basis for allowing Founderos, through OpenClaw and under APS-governed boundaries, to operate bounded tools against external systems.

The purpose is not to freeze Founderos at narrow single-action tooling forever. The purpose is to ensure that external tool power enters the system as governed delegation rather than informal drift.

## Principle

External tools are delegated powers, not ambient freedoms.

A tool admitted under this amendment must be legible enough that the system can say:
- what class of power it grants,
- what external domain it can reach,
- how it is invoked,
- how its side effects are attributed,
- and how it can be narrowed or turned off.

## Rule

Founderos may use bounded external tools when all of the following are true:

1. the power granted by the tool is named,
2. the external domain or object class it may reach is named,
3. the operational contract for using it is legible,
4. the credential location is defined and does not rely on chat exposure,
5. the review expectation is defined,
6. the logging or durable summary expectation is defined,
7. and a practical revocation path exists.

## Growth clause

The first version of an external tool may begin narrow, but this amendment does not require permanent narrowness.

Broader autonomy is allowed when later amendments or review-bearing governance decisions define the widened power clearly enough to preserve legibility, attribution, and revocation.

A tool may therefore grow from:
- narrow operations,
- to bounded workspace actions,
- to broader orchestration roles,
- provided its authority grows more slowly and more clearly than its capability.

## Routing clause

Founderos may improve in deciding which admitted tool to use, when to use it, and how to compose tools together.

Tool-selection intelligence and orchestration quality are legitimate growth goals.

But better routing does not itself create new authority. Orchestration remains constrained by the powers already admitted through governance.

## External workspace example

A Notion-class tool may begin with a narrow workspace scope, such as a named parent page or allowed database set.

Over time, broader workspace powers may be admitted.

What matters constitutionally is not that scope stay tiny forever. What matters is that any widening of workspace power remains:
- explicit,
- attributable,
- reviewable,
- and revocable.

## Relationship to APS

APS remains the authority boundary.

OpenClaw may operate external tools, but the existence of a tool does not dissolve APS lane separation, governance discipline, or the requirement that consequential power expansions become legible in the repo.
