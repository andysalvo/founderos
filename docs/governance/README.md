# Governance

This directory contains the formal governance layer for Founderos.

It is not supplemental reading.
It is part of the system.

A reader should not assume they fully understand Founderos from the product description, API surface, worker wiring, or deployment docs alone. Those explain how the system runs. The governance layer explains what the system is permitted to become, how authority is structured, how powers may expand, and what must remain bounded even as technology improves.

This directory should be treated as required reading before:

- adding OpenClaw tools,
- widening APS authority,
- introducing external workspace automation,
- broadening execution autonomy,
- or interpreting Founderos as a general autonomous agent.

## Reading order

1. `PRE_CONSTITUTION_RESEARCH_CORPUS.md`
2. `CONSTITUTION.md`
3. `amendments/README.md`
4. active amendments in numerical order
5. relevant governance-bearing entries in `memory/decisions/`

## What this directory is for

This directory exists so Founderos can grow without becoming vague about power.

The architecture layer explains:
- what interfaces and services exist,
- how APS, OpenClaw, GitHub, and Supabase relate,
- and where execution flows run.

The governance layer explains:
- who holds authority,
- what AI may and may not do,
- how new powers enter the system,
- how powers widen over time,
- and how powers can be narrowed again.

## Core claim

Founderos is not fully legible without its governance layer.

A reader should be able to answer all of the following from this corpus:
- What is Founderos?
- Who has constitutional authority?
- What role does APS play?
- What role does OpenClaw play?
- How does capability growth happen?
- When does a tool change become amendment-level?
- How can delegated powers be reviewed, suspended, or revoked?
